## 1. Headless 组件：逻辑与样式的彻底解耦

### 核心思想
Headless 组件只负责管理状态、行为和可访问性，完全不定义任何 UI 样式。它通过作用域插槽或 expose 将内部状态暴露给消费者，由消费者完全控制渲染内容。这种模式让同一套交互逻辑可以适配完全不同的视觉设计。

### 适用场景
- 需要支持多种主题/皮肤的设计系统（如 Tabs、Dialog、Dropdown）
- 跨项目复用复杂交互逻辑，但各项目 UI 风格差异大
- 构建底层组件库时，希望上层团队拥有完全的样式自由度

### 代码示例

```vue
<!-- HeadlessTabs.vue -->
<script setup lang="ts">
import { ref, computed, provide, inject } from 'vue'

const props = defineProps<{
  modelValue: string
}>()

const emit = defineEmits<{
  'update:modelValue': [value: string]
}>()

const activeTab = computed({
  get: () => props.modelValue,
  set: (v) => emit('update:modelValue', v)
})

const tabs = ref<string[]>([])

function registerTab(id: string) {
  if (!tabs.value.includes(id)) tabs.value.push(id)
}

function setActive(id: string) {
  activeTab.value = id
}

provide('tabs', { activeTab, registerTab, setActive })
</script>

<template>
  <slot
    :tabs="tabs"
    :active-tab="activeTab"
    :set-active="setActive"
  />
</template>
```

```vue
<!-- HeadlessTabPanel.vue -->
<script setup lang="ts">
import { inject, onMounted } from 'vue'

const props = defineProps<{ id: string }>()
const tabs = inject<any>('tabs')

onMounted(() => tabs?.registerTab(props.id))
</script>

<template>
  <slot :is-active="tabs?.activeTab === props.id" />
</template>
```

**消费者完全控制样式：**

```vue
<script setup lang="ts">
import { ref } from 'vue'
import HeadlessTabs from './HeadlessTabs.vue'
import HeadlessTabPanel from './HeadlessTabPanel.vue'

const active = ref('tab1')
</script>

<template>
  <HeadlessTabs v-model="active" v-slot="{ tabs, activeTab, setActive }">
    <div class="tabs">
      <button
        v-for="tab in tabs"
        :key="tab"
        :class="['tab', { active: activeTab === tab }]"
        @click="setActive(tab)"
      >
        {{ tab }}
      </button>
    </div>

    <HeadlessTabPanel id="tab1" v-slot="{ isActive }">
      <div v-show="isActive" class="panel">内容一</div>
    </HeadlessTabPanel>
    <HeadlessTabPanel id="tab2" v-slot="{ isActive }">
      <div v-show="isActive" class="panel">内容二</div>
    </HeadlessTabPanel>
  </HeadlessTabs>
</template>

<style scoped>
.tab.active { border-bottom: 2px solid blue; color: blue; }
.panel { padding: 16px; }
</style>
```

### 注意事项
- 必须提供完整的可访问性支持（ARIA 属性、键盘导航），因为消费者不会自行处理
- 状态暴露的粒度要适中：太粗导致消费者不灵活，太细增加使用复杂度
- 建议配套提供"默认样式版本"和 Headless 版本，满足不同层级需求
- 注意 provide/inject 的 TypeScript 类型安全，建议定义接口而非用 `any`

## 2. Composable 的依赖注入模式：可测试的状态共享

### 核心思想
将组件间共享的状态和逻辑封装为 Composable，并通过参数注入依赖（而非直接引用全局状态），使 Composable 在保持复用性的同时具备良好的可测试性和可组合性。

### 适用场景
- 需要在多个组件间共享状态，但又不希望硬耦合到 Pinia/Vuex
- 编写可复用的业务逻辑 Composable（如表单处理、列表筛选）
- 单元测试时需要 mock 依赖的场景

### 代码示例

```ts
import type { Ref } from 'vue'
// useFilterableList.ts
import { computed, ref } from 'vue'

interface Item { id: string, name: string, category: string }

export interface FilterableListDeps {
  fetchItems: (filters: Record<string, any>) => Promise<Item[]>
  debounceMs?: number
}

export function useFilterableList(deps: FilterableListDeps) {
  const items = ref<Item[]>([])
  const loading = ref(false)
  const error = ref<string | null>(null)
  const filters = ref<Record<string, any>>({})

  let debounceTimer: ReturnType<typeof setTimeout>

  const filteredItems = computed(() => {
    // 本地过滤逻辑...
    return items.value
  })

  async function load() {
    loading.value = true
    error.value = null
    try {
      items.value = await deps.fetchItems(filters.value)
    }
    catch (e) {
      error.value = e instanceof Error ? e.message : '加载失败'
    }
    finally {
      loading.value = false
    }
  }

  function setFilters(newFilters: Record<string, any>) {
    filters.value = newFilters
    clearTimeout(debounceTimer)
    debounceTimer = setTimeout(load, deps.debounceMs ?? 300)
  }

  return {
    items,
    loading,
    error,
    filters,
    filteredItems,
    load,
    setFilters,
  }
}
```

```vue
<!-- ProductList.vue -->
<script setup lang="ts">
import { useFilterableList } from './useFilterableList'
import { productApi } from '@/api'

const { items, loading, setFilters } = useFilterableList({
  fetchItems: productApi.getList,
  debounceMs: 500,
})
</script>
```

**测试时轻松 mock：**

```ts
import { describe, expect, it, vi } from 'vitest'
import { useFilterableList } from './useFilterableList'

it('应该根据过滤条件加载数据', async () => {
  const mockFetch = vi.fn().mockResolvedValue([{ id: '1', name: 'A', category: 'x' }])
  const { load, items } = useFilterableList({ fetchItems: mockFetch })

  await load()

  expect(mockFetch).toHaveBeenCalledWith({})
  expect(items.value).toHaveLength(1)
})
```

### 注意事项
- 依赖注入的粒度要合理：通用依赖（如 `fetch`）通过参数传入，通用工具（如 `ref`）直接使用
- 避免"依赖地狱"：如果 Composable 需要超过 3-4 个依赖，考虑拆分或封装配置对象
- 为依赖定义接口（`FilterableListDeps`），便于 TypeScript 类型检查和 mock
- 注意 Composable 内部创建的响应式数据在多次调用时的独立性

## 3. 受控与非受控的混合模式：灵活的表单组件

### 核心思想
组件同时支持受控模式（外部通过 `v-model` 完全控制状态）和非受控模式（组件内部自行管理状态）。通过检测 `modelValue` 是否存在来决定当前工作模式，让组件在不同场景下都能提供最佳开发体验。

### 适用场景
- 设计系统的基础表单组件（Input、Select、Switch 等）
- 需要快速原型开发时（非受控模式减少样板代码）
- 复杂表单场景需要外部精确控制时（受控模式）

### 代码示例

```vue
<!-- HybridInput.vue -->
<script setup lang="ts">
import { ref, computed, watch } from 'vue'

const props = defineProps<{
  modelValue?: string
}>()

const emit = defineEmits<{
  'update:modelValue': [value: string]
}>()

// 判断是否处于受控模式
const isControlled = computed(() => props.modelValue !== undefined)

// 内部状态（非受控模式使用）
const internalValue = ref('')

// 统一对外暴露的值
const value = computed({
  get: () => (isControlled.value ? props.modelValue! : internalValue.value),
  set: (val) => {
    if (isControlled.value) {
      emit('update:modelValue', val)
    } else {
      internalValue.value = val
    }
  }
})

// 受控模式下，同步外部值到内部（用于 reset 等场景）
watch(() => props.modelValue, (val) => {
  if (isControlled.value && val !== undefined) {
    internalValue.value = val
  }
})
</script>

<template>
  <input
    v-model="value"
    class="hybrid-input"
    v-bind="$attrs"
  />
</template>
```

**非受控模式（简洁）：**

```vue
<HybridInput placeholder="请输入" />
```

**受控模式（精确控制）：**

```vue
<script setup lang="ts">
import { ref } from 'vue'
const name = ref('')
</script>

<template>
  <HybridInput v-model="name" placeholder="请输入姓名" />
  <p>当前值：{{ name }}</p>
</template>
```

### 注意事项
- 必须明确区分两种模式，避免"半受控"状态导致数据不同步的 bug
- 非受控模式下，如果需要获取当前值，可通过 `ref` + `expose` 暴露内部状态
- 表单校验场景下，受控模式更易于实现实时校验和错误提示
- 建议文档中明确说明两种模式的使用方式和切换条件

## 4. 组件 API 的渐进式暴露：从简单到强大

### 核心思想
组件的公共 API 应该分层设计：基础用法极其简单（一个 `v-model` 搞定），高级用法通过额外的 props/slots/events 逐步解锁。避免为了让组件"强大"而强迫所有用户都面对复杂的配置。

### 适用场景
- 设计系统中使用频率极高的基础组件（Button、Input、Modal 等）
- 需要同时服务"快速开发"和"精细定制"两种用户群体的组件
- 组件库的版本迭代中，需要保持向后兼容的同时增加新能力

### 代码示例

```vue
<!-- SmartModal.vue — 渐进式 API 设计 -->
<script setup lang="ts">
import { computed } from 'vue'

interface Props {
  // 基础层：最简单的用法只需要这个
  modelValue: boolean

  // 渐进层：按需增加的常用配置
  title?: string
  width?: string
  closable?: boolean

  // 高级层：精细控制
  appendToBody?: boolean
  destroyOnClose?: boolean
  beforeClose?: (done: () => void) => void
}

const props = withDefaults(defineProps<Props>(), {
  width: '520px',
  closable: true,
  appendToBody: false,
  destroyOnClose: false,
})

const emit = defineEmits<{
  'update:modelValue': [boolean]
  open: []
  close: []
  closed: []  // 动画结束后
}>()

const visible = computed({
  get: () => props.modelValue,
  set: (v) => emit('update:modelValue', v)
})

function handleClose() {
  if (props.beforeClose) {
    props.beforeClose(() => { visible.value = false })
  } else {
    visible.value = false
  }
}
</script>

<template>
  <Teleport to="body" :disabled="!appendToBody">
    <Transition name="modal">
      <div v-if="visible" class="modal-mask" @click.self="handleClose">
        <div class="modal-container" :style="{ width }">
          <!-- 标题栏：可通过 slot 完全自定义，也可只传 title prop -->
          <div v-if="$slots.header || title" class="modal-header">
            <slot name="header" :close="handleClose">
              <h3>{{ title }}</h3>
              <button v-if="closable" @click="handleClose">×</button>
            </slot>
          </div>

          <!-- 内容区 -->
          <div class="modal-body">
            <slot />
          </div>

          <!-- 底部：默认不渲染，传入 footer slot 或特定 prop 时显示 -->
          <div v-if="$slots.footer" class="modal-footer">
            <slot name="footer" :close="handleClose" />
          </div>
        </div>
      </div>
    </Transition>
  </Teleport>
</template>
```

**基础用法：**

```vue
<SmartModal v-model="visible" title="提示">
  确定要删除吗？
</SmartModal>
```

**渐进用法：**

```vue
<SmartModal
  v-model="visible"
  title="编辑"
  width="800px"
  :before-close="checkUnsaved"
>
  <EditForm />
  <template #footer="{ close }">
    <button @click="close">取消</button>
    <button @click="save">保存</button>
  </template>
</SmartModal>
```

### 注意事项
- 默认值的选择要符合"最小惊讶原则"：最常见的场景应该零配置就能工作
- 高级 props 的命名要语义化（如 `beforeClose` 比 `onBeforeClose` 更符合 Vue 惯例）
- Slot 的作用域参数设计要完整，让消费者能完全替代默认行为
- 避免 props 之间的隐式冲突（如同时传 `title` 和 `#header` 时，slot 应该优先）

## 5. 微前端中的组件共享：基于 Module Federation 的跨应用复用

### 核心思想
在微前端架构下，通过 Module Federation 将组件作为独立的远程模块暴露，让多个独立部署的应用能够共享同一套组件实现，同时保持各自的技术栈独立性和部署灵活性。

### 适用场景
- 大型前端应用拆分为多个独立微应用，需要共享通用组件库
- 多团队并行开发，希望避免组件库版本碎片化
- 需要渐进式迁移技术栈（如 Vue 2 到 Vue 3）时共享组件

### 代码示例

**组件库应用（远程提供者）：**

```ts
// vue.config.ts — 组件库配置
import { defineConfig } from '@vue/cli-service'
import { ModuleFederationPlugin } from 'webpack'

export default defineConfig({
  configureWebpack: {
    plugins: [
      new ModuleFederationPlugin({
        name: 'shared_components',
        filename: 'remoteEntry.js',
        exposes: {
          './DataTable': './src/components/DataTable/index.vue',
          './SearchForm': './src/components/SearchForm/index.vue',
          './usePermission': './src/composables/usePermission.ts',
        },
        shared: {
          'vue': { singleton: true, eager: false, requiredVersion: '^3.0.0' },
          'element-plus': { singleton: true, eager: false },
        },
      }),
    ],
  },
})
```

**业务应用（消费者）：**

```ts
// main.ts — 动态加载远程组件
import { createApp, defineAsyncComponent } from 'vue'
import App from './App.vue'

// 声明远程模块类型（TypeScript 支持）
declare module 'shared_components/DataTable' {
  import { Component } from 'vue'

  const component: Component
  export default component
}

const app = createApp(App)

// 注册远程组件
const RemoteDataTable = defineAsyncComponent(
  () => import('shared_components/DataTable')
)

app.component('RemoteDataTable', RemoteDataTable)
app.mount('#app')
```

```vue
<!-- 业务应用中使用 -->
<script setup lang="ts">
import { ref } from 'vue'

const tableData = ref([
  { id: 1, name: '产品 A', status: 'active' },
  { id: 2, name: '产品 B', status: 'inactive' },
])
</script>

<template>
  <RemoteDataTable
    :data="tableData"
    :columns="[
      { prop: 'name', label: '名称' },
      { prop: 'status', label: '状态' },
    ]"
  >
    <template #status="{ row }">
      <span :class="row.status">{{ row.status }}</span>
    </template>
  </RemoteDataTable>
</template>
```

### 注意事项
- `singleton: true` 确保 Vue 实例在多个微应用间共享，避免响应式系统冲突
- 远程组件的加载需要处理 loading 和 error 状态，建议配合 `defineAsyncComponent` 的 `loadingComponent` 和 `errorComponent`
- 组件库的版本升级要考虑向后兼容性，建议使用语义化版本控制
- 网络延迟是微前端的固有问题，对高频使用的组件建议做本地缓存或预加载
- TypeScript 类型共享需要额外配置（如 `dts-loader` 或手动声明文件）

---

> 💡 **今日思考**：好的组件 API 设计应该像洋葱——外层简单直接，内层丰富强大。渐进式暴露不仅降低了上手门槛，也让组件在不同复杂度场景下都能保持优雅。记住：每个新增的 prop、slot 或 event 都是在增加使用者的认知负担，务必确保它带来的价值大于成本。

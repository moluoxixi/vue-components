# ADR-0006 默认知识库检索策略（结构化关键词 topK）

## 状态

accepted（2026-06-15）

## 背景

组件库知识库的核心语料来自公开组件契约：组件名、Props、Emits、Slots、Model、关联类型及其字段注释。这类数据天然结构化，用户问题也常直接命中组件名、属性名、事件名、插槽名或业务注释。开发者确认：即便组件数达到几千到上万个，也不必默认引入向量工具库；先用结构化关键词 topK 可以降低依赖、启动成本、token 消耗和可解释性风险。

## 决策

默认 `content` 模式采用结构化关键词 topK 检索：

- 构建时只抽取公共组件契约并渲染为稳定正文，不触发 embedding。
- 检索时按组件名、契约正文、Props/Emits/Slots/Model、类型字段与注释做轻量打分。
- 只把 `score > 0` 且排名前 `topK` 的契约放进 chat 上下文。
- 无关键词命中时返回 empty，由上层输出「未找到相关组件信息」，不把无关组件塞给模型。
- 插件 content 模式可在面板首次打开时自动准备知识库；CLI `build-index` 保留为显式刷新与 CI 校验入口。

## 影响

- 默认链路零 embedding、零向量库、零远端 embedding key。
- `topK` 在 content 与 vector 两种模式下语义一致，都是上下文组件数上限。
- 检索结果可解释，来源分数来自关键词命中而非黑盒向量距离。
- 组件字段注释是知识库质量关键输入，抽取与渲染必须保留描述文本。

## 约束

- content 模式不得回退为全量上下文；否则组件数增加时 token 成本不可控。
- 不得在无命中时随机取首个组件兜底；无依据必须显式暴露。
- 只纳入公开组件契约与关联类型，不把内部实现源码、spike、测试目录或构建产物作为知识库语料。
